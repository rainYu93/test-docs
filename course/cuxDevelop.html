# 1 概述

# 1.1 文档概述

概述

该文档主要介绍移动端基于标准二开的推荐方式及使用方式， 依赖最低内测版本 "srm-mobile-ui": "2.1.20-alpha", "srm-front-app-core": "2.1.11-alpha"

# 2 二开方式介绍

# 2.1 全量二开

前端样式界面结构大改造近乎重构或者基于标准的代码进行部分改造很难进行、实现，后端接口仍为标准套件提供， 将此类需求的代码开发为一个独立的子应用， 存放在CUX目录下， 并为租户配置一个和对应标准子应用命名一致的子应用。 点击跳转-标准二开子应用开发

# 2.2 局部二开-单页面二开

主体功能保持一致，单个页面界面做了较大的个性化调整或者逻辑的修改

# 2.2.1、开发约定

定义单页面二开的代码存放路径为modules/CUX_PART/page；modules/CUX_PART/page/routers.js 存放覆盖标准页面的二开页面路由配置，meta对象下新增tenantNum、coverPath表示当前二开页面路由在对应的租户下所覆盖的标准路由；modules/CUX_PART/page/xxx.vue为对标准页面进行覆盖的二开页面，我们约定命名二开页面名称的规则为租户+页面文件名.vue的。eg：ApprovalList.vue 甄云租户需要二开，我们定义二开页面名称为ZhenyunApprovalList.vue 。参考下图及代码：

点击查看routers.js代码 
let route = [
  {
    path: "/ZhenyunApprovalList",

  }

export default route;

# 2.3 局部二开-方法二开

所有代码几乎和标准保持一致， 部分界面的某个方法逻辑进行个性化定制， 我们提供标准方法根据租户进行覆盖执行二开逻辑

# 2.2.1、开发约定

定义存放二开方法的代码路径为modules/CUX_PART/Method；定义文件modules/CUX_PART/Method/xxx.js, 二开js文件名我们约定以标准方法所存在的标准文件名+Method.js结尾；eg: 我们覆盖审批列表页ApprovalList.vue中的toDetail方法，我们需要命名的二开js文件名应该为ApprovalListMethod.js。文件内存放二开方法（我们约定二开方法的命名规则，租户编码+原方法名称转驼峰）及租户下标准方法与二开方法的映射关系（我们约定租户编码作位一个配置对象的区分表示，对象中key为原标准方法名称，value为二开的方法对象）, js代码中export default {}内容直接复制即可，所有Method.js保持一致。参考下图及代码：

点击查看xxxMethod.js代码 
import SrmAppCore from "srm-front-app-core";

//为啥不放在methods里面？ 如果所有的二开方法都写入methods，也就是说即使这个二开方法不使用，仍然会全部被加载到当前实例里面，会增大性能消耗，放在此处
// 需要的时候，在以约定的命名规则动态挂载的当前实例，以供使用；方法命名规则租户编码+原方法名称 驼峰
function zhenYunToDetail(vm) {
  console.info("I can get all information about ApprovalList instance", vm.statusMap, vm);
  vm.$dialog({
    content:"Hi, method toDetail in ApprovalList had been cux. See something in console"
  });
}

//定义方法映射关系
const cuxFunctions = {
  "SRM-ZHENYUN1": { //⼆开租户的编码
    "toDetail": zhenYunToDetail, //key 标准方法名称，value二开方法
  }
};

export default {
  data() {
    return {
    };
  },
  beforeCreate() {
    //抽取一个通用方法，规则构建基于标准二开的埋点方法
    SrmAppCore.buildCuxFunction(this, cuxFunctions);
  },
};

# 2.2.2、使用方式

标准方法下进行埋点，我们约定埋点的方法名以over$+原方名为命名规则,eg:标准方法为toDetail,埋点方法为over$toDetail，需要埋点的方法都按照下图的格式写在原方法第一行， 每个方法只需要埋点一次，不同方法间埋点的差异只是埋点方法名称的不同。如下图：

点击查看埋点代码 
 // 标准二开埋点
      if (this.over$toDetail && typeof this.over$toDetail === "function") {
        const response = this.over$toDetail();
        return response;
      }

# 2.4 局部二开-组件二开

所有代码几乎和标准保持一致， 部分组件样式个性化， 我们提供标准方法根据租户进行替换标准组件为二开组件的操作。

# 2.2.1、开发约定

定义存放二开组件的代码路径为modules/CUX_PART/Component；modules/CUX_PART/Component/config.js存放标准、二开组件的映射关系（我们约定租户编码作为一个配置对象的区分表示，对象中key为原标准组件名称(并非指组件文件名称，而是组件内定义的name值。)，value为二开的组件对象）； 定义二开组件文件modules/CUX_PART/Component/xxx.vue ，文件名我们约定以租户+原组件名.vue的规则定义。eg：ApprovalPopup.vue 甄云租户需要二开， 我们定义二开组件名称为ZhenyunApprovalPopup.vue。具体如下图：

点击查看config.js代码 
import ZhenyunApprovalPopup from "./ZhenyunApprovalPopup";
export default {
  "SRM-ZHENYUN1": { //⼆开租户的编码
    "ApprovalPopup": ZhenyunApprovalPopup, //key 标准组件名称，value二开组件对象
  }
};
